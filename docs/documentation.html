Design Philosophy

Project 1 Approach

Use the ecosystem. LangChain handles the retrieval chain, EnsembleRetriever fuses BM25 + HNSW, ChromaDB manages the vector index. The focus is on wiring well-tested components together and adding dual-layer memory on top.

Project 2 Approach

Build from scratch. Every algorithm (HNSW, BM25, RRF, Self-RAG, compression) is implemented in pure Python. The focus is on full transparency, learnability, and control over every step.

RAG LangChain Chroma

Production-ready conversational document assistant · FastAPI · LangChain · ChromaDB · Port 8000

Intent & Purpose

This project demonstrates a production-quality RAG stack assembled from well-established ecosystem components. The intent is to show how LangChain, ChromaDB, and BM25 can be combined with a dual-layer memory system to build a document assistant that maintains conversational continuity — both within a session and across separate sessions.

Key design decision: Use LangChain's EnsembleRetriever to fuse HNSW vector search and BM25 keyword search in a single retrieval call, without implementing fusion logic manually.

Memory System

In-Chat (Session) Memory

Each session maintains an isolated message history. The last 20 user/assistant turn pairs are injected into every LangChain prompt as MessagesPlaceholder, giving the LLM full conversational context. Sessions survive server restarts via data/sessions.json.

Cross-Chat Memory

When a session is deleted or archived:

  1. Full transcript is sent to the LLM with a summarisation prompt → 200–400 word dense summary.
  2. Summary is embedded with text-embedding-3-small.
  3. Stored/upserted in the chat_memory ChromaDB collection.
  4. On every new query, top-3 semantically similar summaries are retrieved and injected into the system prompt.

Hybrid Retrieval

RetrieverMechanismStrength
Vector (HNSW)ANN search on OpenAI embeddings via ChromaDBSemantic similarity; paraphrase matching
BM25Keyword TF-IDF ranking via rank-bm25 in memoryExact term matching; rare words

Results are merged using LangChain's EnsembleRetriever with configurable weights (default 50/50) and deduplicated.

RAG Custom Engine

Complete RAG pipeline from scratch — No LangChain · No ChromaDB · Pure Python · Port 8001

Intent & Purpose

This project proves that every component of a modern RAG system can be implemented in pure Python with full algorithmic transparency. It is a learning-oriented reference implementation that trades ecosystem convenience for visibility into every step of the retrieval and generation process.

Philosophy: If LangChain's EnsembleRetriever is a black box, this project opens the box — showing exactly how score normalisation, RRF, and Self-RAG grading work at the code level.

Key Features

FeatureDescription
Custom HNSW Vector StorePure-Python HNSW graph + brute-force cosine fallback; JSON persistence; thread-safe singleton
Custom BM25 IndexOkapi BM25 from scratch; add/remove by metadata; JSON persistence
Hybrid RetrievalWeighted score ensemble and Reciprocal Rank Fusion (RRF) modes; selectable per request
Multi-Query RetrieverLLM generates N alternative phrasings; all variants searched; results merged via RRF
Self-RAG3-stage LLM gates: retrieval decision → relevance grading → hallucination check
Contextual CompressionLLM extracts only query-relevant sentences from each retrieved chunk
Pipeline OrchestratorStep-by-step trace with timing; async SSE streaming to frontend; trace stored per session
Dual-Layer MemoryIn-chat session history + cross-session LLM summarisation with vector search

Architecture

Pipeline Steps

The orchestrator runs up to 8 pipeline steps per query. Each step is configurable and can be enabled/disabled per request:

1. Cross-Session Memory Search
→
2. Retrieval Decision (Self-RAG)
→
3. Multi-Query Expansion
→
4. Hybrid Search
→
5. Relevance Grading (Self-RAG)
→
6. Contextual Compression
→
7. Answer Generation
→
8. Hallucination Check (Self-RAG)

Self-RAG (Adaptive Retrieval)

Stage 1 — Retrieval Decision

Before any retrieval, the LLM evaluates whether the query requires document lookup. Greetings, meta questions, and simple arithmetic skip retrieval entirely and go straight to generation, saving latency and API cost.

Stage 2 — Relevance Grading

After retrieval, each chunk is individually graded by the LLM for relevance to the query. Chunks marked not_relevant are filtered out before the answer is generated, reducing hallucination risk from noisy context.

Stage 3 — Hallucination Check

After generation, the LLM verifies that every factual claim in the answer is grounded in the provided context. Ungrounded answers are flagged in the pipeline trace.

Multi-Query Retriever

Generates N alternative phrasings of the user's query, runs hybrid search for each, then merges all result lists via RRF. A document appearing in multiple result sets receives a compounded rank score, improving recall for queries that benefit from diverse phrasing.

Contextual Compression

For each retrieved chunk (≥ 200 characters), the LLM extracts only the sentences directly relevant to the query. This reduces the context window size while retaining key information. Compression statistics (original length, compressed length, ratio) are tracked in the pipeline trace per chunk.

Hybrid Retrieval Modes

ModeMechanismWhen to use
Weighted EnsembleMin-max normalise both score lists; combine: score = v×w_v + b×w_bWhen retriever quality is predictable and weights can be tuned
RRFPosition-based: score = Σ 1/(60 + rank_i)When score scales differ between retrievers; more robust default

Use RAG LangChain Chroma when…

  • You need a production-ready document assistant quickly
  • You want to build on well-maintained ecosystem components
  • Your team already knows LangChain

Deficiencies & Roadmap

Honest assessment of current limitations across both projects, and suggested improvements.

Legend: HIGH Significant limitation affecting production readiness. MEDIUM Affects performance or cost in common scenarios. LOW Nice-to-have improvements.

Shared Deficiencies (Both Projects)

SeverityAreaIssueSuggested Fix
HIGH Authentication No authentication or authorisation layer. Any client with network access can read, upload, or delete all data. Add API key middleware or OAuth2 with FastAPI security utilities.
HIGH Answer Streaming Answers are not streamed token-by-token. The full generation completes before the response is sent, creating latency spikes on long answers. Use OpenAI stream=True with async generators and SSE in both projects.
MEDIUM Pagination GET /api/documents and GET /api/sessions return all records without pagination. Large collections will cause slow responses. Add ?limit=N&offset=M query parameters to list endpoints.
MEDIUM Embedding Concurrency Document embeddings are computed in a synchronous batch in the request handler thread. Large documents block the event loop. Wrap embedding calls in run_in_executor or use a background task queue (e.g., Celery, ARQ).
MEDIUM Error Handling OpenAI API failures (rate limits, timeouts) bubble up as 500 errors with no retry logic. Add exponential back-off retry using tenacity around all OpenAI calls.
LOW No Re-ranking No cross-encoder or neural re-ranking step after retrieval. Final ranking relies entirely on BM25 + vector fusion. Integrate a cross-encoder model (e.g., cross-encoder/ms-marco-MiniLM via sentence-transformers) as an optional re-ranking step.
LOW No Evaluation No automated evaluation of retrieval precision/recall or answer quality (RAGAS, TruLens). Integrate RAGAS for faithfulness, answer relevancy, and context precision metrics.

RAG LangChain Chroma — Specific Issues

SeverityAreaIssueSuggested Fix
MEDIUM No Advanced Retrieval No multi-query expansion, Self-RAG, or contextual compression. Single-shot retrieval with a fixed query. Port the multi-query and compression components from the Custom Engine, or use LangChain's MultiQueryRetriever.
MEDIUM ChromaDB Persistence ChromaDB stores data in a local directory; not suitable for multi-instance deployments without a shared filesystem or a hosted ChromaDB server. Configure ChromaDB in client-server mode, or migrate to a hosted vector DB (Pinecone, Weaviate, Qdrant).
LOW No Pipeline Trace No visibility into retrieval scores, chunk counts, or timing per request. Debugging is harder. Add a debug flag to the query endpoint that returns retrieval metadata.
LOW LangChain Version Coupling LangChain's API changes frequently; the project pins ecosystem packages that may drift out of compatibility. Pin exact versions in requirements.txt and add a CI check for dependency conflicts.

RAG Custom Engine — Specific Issues

SeverityAreaIssueSuggested Fix
HIGH JSON Index Scalability All indexes (vector store, BM25) are loaded fully into memory and serialised to JSON. Does not scale beyond tens of thousands of chunks; will OOM on large corpora. Replace JSON persistence with FAISS or Annoy for vectors and SQLite for BM25; load lazily on query.
HIGH HNSW Implementation Pedagogical HNSW — lacks proper multi-layer graph construction, ef-search tuning, and node deletion. The brute-force fallback is O(n) per query. Use FAISS (HNSW index) or hnswlib for production-grade HNSW with proper recall guarantees.
MEDIUM Self-RAG API Cost With use_self_rag=true, every query triggers 2–3 additional LLM calls (retrieval decision + per-chunk grading + hallucination check), multiplying API cost and latency by 3–5×. Cache retrieval decision results by query hash; batch chunk grading into a single LLM call.
MEDIUM Multi-Query Cost use_multi_query=true triggers 1 extra LLM call for variant generation plus N×hybrid_search calls. Significantly increases latency (often 3–8 seconds extra). Limit to N=2 variants by default; add a latency budget parameter.
MEDIUM Compression Cost use_compression=true triggers one LLM call per retrieved chunk (often 5 chunks → 5 calls). Adds 5–10 seconds to query latency. Batch all chunks into a single LLM call with a structured output format; skip compression for chunks < 300 characters.
MEDIUM JSON Write Concurrency Simple threading lock flushing the entire JSON file on every write. High-concurrency scenarios will see lock contention. Use SQLite (WAL mode) for concurrent write safety, or an async lock with debounced flush.
LOW No Answer Streaming The SSE endpoint streams pipeline step events but the answer text arrives as a single event after full generation. No token-by-token streaming. Use OpenAI stream=True and emit token chunks as SSE events after all steps complete.
